# Issue #9: Copying Markdown Strips Code Fence Markers

**Date:** 2026-03-23
**Status:** Investigation complete — hypothesis disproven for text/plain; real risk identified for text/html (rich paste)

## Problem Statement

Users report that copying markdown from the editor strips code fence markers
(triple backticks) from the clipboard. The hypothesis is that the decoration
system's character-hiding technique (`color: transparent` + `letterSpacing:
'-1em'`) affects what VS Code puts on the clipboard.

## Investigation Summary

### Finding 1: Decorations Do NOT Affect text/plain Clipboard

**The original hypothesis is almost certainly wrong for plain-text paste.**

VS Code's copy command reads from the **text model** (the document buffer),
not from the view layer. Decorations are purely visual — they change how text
renders in the editor but do not modify the underlying document content.
The `TextEditorDecorationType` API is documented as a visual-only mechanism.

When a user presses Cmd+C, VS Code calls `model.getValueInRange()` on the
selected range. This reads directly from the text buffer. The decoration
system — including `color: transparent`, `letterSpacing: '-1em'`, `opacity`,
`textDecoration`, etc. — has zero influence on what `getValueInRange()`
returns.

**Evidence:**
- The vscode-inline-fold extension uses the identical `letterSpacing: '-1em'`
  + `color: transparent` technique to visually hide Tailwind CSS classes.
  No clipboard stripping issues are reported in its issue tracker.
- VS Code issue #13632 ("Hide character sequences through decorators") and
  issue #45393 ("API for hiding characters") discuss decoration-based hiding
  extensively. Neither mentions clipboard side effects.
- The VS Code decoration sample (`vscode-extension-samples/decorator-sample`)
  confirms decorations are visual only.

**Conclusion:** If a user copies text in VS Code and pastes into a plain-text
target (another editor, terminal, Cmd+V into VS Code itself), all characters
including hidden-decorated ones will be present. The code fence markers
(`\`\`\``) are in the document and will be on the clipboard.

### Finding 2: text/html Clipboard Format IS a Risk Vector

When `editor.copyWithSyntaxHighlighting` is enabled (the default), VS Code
writes **two formats** to the clipboard:

1. `text/plain` — raw text from the model (unaffected by decorations)
2. `text/html` — HTML with inline styles derived from TextMate tokenization

The HTML format is generated from **TextMate scopes**, not from extension
decorations. Extension `TextEditorDecorationType` styles are NOT included
in the HTML clipboard format. So even the rich-text copy should not strip
characters.

However, there is a nuanced scenario worth investigating:

**If a user copies from VS Code and pastes into a rich-text target** (Google
Docs, Slack, email composer), the target application chooses `text/html`
over `text/plain`. The HTML generated by VS Code's tokenizer should include
the code fence characters — but the rendering in the target app could
potentially collapse or hide them if the HTML structure is unusual.

This is NOT caused by our extension's decorations, but it could explain the
user-reported symptom if they are pasting into a rich-text destination.

### Finding 3: What Our Extension Actually Hides

The `HIDDEN_COLLAPSED` style is used for these marker types:

| Provider | What it hides | File |
|---|---|---|
| `HeadingMarkersProvider` | `# `, `## `, etc. | markdown-polish.ts:106 |
| `BoldItalicMarkersProvider` | `*`, `**`, `***` | markdown-polish.ts:1144 |
| `LinkMarkersProvider` | `[`, `](`, URL, `)` | markdown-polish.ts:1151 |
| `InlineCodeMarkersProvider` | `` ` ``, ` `` ` ` | markdown-polish.ts:1165 |
| `DelimiterProvider` (CriticMarkup) | `{++`, `++}`, `{--`, `--}`, etc. | criticmarkup.ts:489 |
| `SubstitutionArrowProvider` | `~>` | criticmarkup.ts:520 |
| `SubstitutionOldProvider` | Old text in `{~~ old ~> new ~~}` | criticmarkup.ts:63 |

**Code block fence lines** (the `` ``` `` lines themselves) are NOT hidden with
`letterSpacing: '-1em'`. They use **opacity dimming** instead:

```typescript
// markdown-polish.ts, lines 197-203
const CODE_BLOCK_FENCE_COLLAPSED: vscode.DecorationRenderOptions = {
    opacity: CODE_BLOCK_FENCE_DIM_OPACITY,  // '0.4'
};
const CODE_BLOCK_FENCE_EXPANDED: vscode.DecorationRenderOptions = {
    opacity: '1.0',
};
```

This means the code fence markers are dimmed to 40% opacity when the cursor
is away, but they are NOT hidden with the `letterSpacing` technique. The
`opacity` property has even less potential to affect clipboard than
`letterSpacing` — it is purely a rendering property.

**Frontmatter delimiters** (`` ``` `` at top of file) are also dimmed via
opacity, not hidden:

```typescript
// markdown-polish.ts, lines 197-198 (same fence provider covers both)
// Frontmatter ``` lines are detected and skipped by detectFrontmatterEnd(),
// while the FrontmatterProvider dims the entire block with opacity.
```

### Finding 4: Reproducing the Issue

Given that decorations cannot strip text from the clipboard, the reported
issue likely has a different root cause. Possible explanations:

1. **User is pasting into a rich-text app** and the target app is rendering
   the pasted content in a way that hides or collapses the backticks. This
   would look like "copy stripped the backticks" but is actually a rendering
   issue in the paste target.

2. **Selection did not include the fence lines.** If the user visually selects
   text inside a code block (where the fences are dimmed to 40% opacity), they
   might not realize their selection does not extend to the fence lines. The
   dimmed appearance could make the fence lines look like they are "part of
   the selection" when they are not.

3. **The user is using the keyboard shortcut to select "inside" the block**
   (e.g., VS Code's "Expand Selection" / Smart Select), which may select
   content between fences but not the fences themselves.

4. **A different extension** is intercepting the copy command or modifying
   clipboard content.

## Recommendations

### Immediate (to close the issue)

1. **Ask the reporter to reproduce with steps.** Specifically:
   - What are they pasting into? (plain text editor vs. rich text app)
   - Are they selecting the fence lines explicitly, or selecting "inside"?
   - Does the issue reproduce with all other extensions disabled?

2. **Verify with F5 testing.** In the Extension Development Host:
   - Open a markdown file with a fenced code block
   - Select text including the ``` lines (confirm they are in the selection)
   - Paste into a plain-text editor (TextEdit in plain mode, or terminal)
   - Verify the backticks are present
   - Then paste into Google Docs / Slack and check if they appear

### If the issue IS real (selection confusion)

3. **Consider making fence dimming less aggressive.** The current opacity of
   `0.4` makes fences quite faint. Increasing to `0.55` or `0.6` would make
   them more visible and reduce the chance of users accidentally excluding
   them from selections.

4. **Add a "Select Code Block" command** (future enhancement) that selects an
   entire fenced code block including its fences, making it easy to copy
   complete blocks.

### If the issue is rich-text paste

5. **This is out of scope for the extension.** VS Code's HTML clipboard
   generation is a core editor feature. The extension cannot control what
   HTML is generated when the user copies.

6. **Document the behavior** — add a note to the extension's documentation
   that copying from VS Code into rich-text apps may not preserve all
   markdown syntax, and recommend pasting as plain text (Cmd+Shift+V in
   most apps).

## VS Code Clipboard Architecture (Reference)

```
User presses Cmd+C
    |
    v
VS Code Copy Handler (clipboard.ts)
    |
    +--> text/plain: model.getValueInRange(selection)
    |       ^-- reads from TEXT MODEL (document buffer)
    |       ^-- decorations have ZERO effect here
    |
    +--> text/html: tokenize(selection) -> inline-styled HTML
    |       ^-- reads from TEXTMATE TOKENIZATION (grammar scopes)
    |       ^-- extension decorations have ZERO effect here
    |
    +--> vscode-editor-data: metadata (language, version)
    |
    v
OS Clipboard (multiple formats stored simultaneously)
    |
    v
Paste target chooses format:
    - Plain text editors -> text/plain (always correct)
    - Rich text apps -> text/html (may render differently)
```

## Key Takeaway

**VS Code decorations are a view-layer concern. They cannot and do not
affect clipboard content.** The `letterSpacing: '-1em'` + `color: transparent`
technique used by this extension (and by vscode-inline-fold, and recommended
in VS Code issue #13632) is safe. The text/plain clipboard format always
contains the exact characters from the document model.

The reported issue is most likely either (a) a selection boundary mistake
exacerbated by dimmed fence lines, or (b) a rich-text paste rendering issue
in the target application. Both are addressable without changing the
decoration system.
